2. 通信のモデル - シャノンとウィーバー

参考文献: クロード・E・シャノン、ワレン・ウィーバー(著)、植松友彦(訳)、通信の数学的理論、ちくま学芸文庫、2009年

2.1 通信システムの構成要素

  • 情報源 → 送信機 → 雑音源 → 受信機 → 受信者
  • 必ずノイズ(雑音)が入るが、実用上問題が無いように見せている

2.2 通信の3つの問題レベル

  1. A) 技術的な問題: 記号をどのくらい正確に伝えることができるのか
  2. B) 意味的な問題: 送信された記号は、どのくらい正確に所望の意図を伝えることができるのか
  3. C) 効果の問題: 受信された意図は、どのくらい効果的に所望する行為に影響するのか
例1: ヴィクトル・ユーゴーと出版社
  • A) ?と!の送信
  • B) 「売れているか?」という問い合わせの意図、「物凄く売れています」という返信の意図
  • C) 作者が売れ行きを気にしている、出版社が喜びを伝えている

2.3 平均情報量(エントロピー)

H = -[p₁log p₁ + p₂log p₂ + ... + pₙlog pₙ]
H = -Σ(i=1 to n) pᵢ log pᵢ

n個の独立な記号の集合が確率pₙで選ばれる場合の平均情報量

5. 技術的課題

5.1 モールス符号の例

重要な違い:
  • 欧文モールス: 使用頻度の高い字に短い符号を割り当てた(E, I, T, S, Aなど)
  • 和文モールス: 欧文をそのままイロハに割り当てた
  • このため通信速度に差が生じる

5.2 通信と放送の違い

  • 1対1: 通信
  • 1対n: 放送
  • 1対n: マルチキャスト通信

5.3 ネットワーク接続の計算

全ノード間接続に必要な線の数:
8個のノード: 7+6+5+4+3+2+1 = 28本
9個: 36本
10個: 45本
一般式: n = (n-1)×n/2

5.4 クロスバースイッチ

  • 全てのノードが同時には通信できないが、線の数は飛躍的に減る
  • 例: 8個のノードで16本(2n)の線で済む

5.5 共有通信路

  • 同時に利用できるのは一対のみと諦めると、nがいくら大きくなっても1本の線で通信できる
  • 無線LANやEthernetの原理
  • 実際には多重化する仕組みで複数ノードが同時通信

5.6 主要な技術的課題

  1. 通信相手を特定すること
  2. 物理的な限界のある通信路を延長すること
  3. 通信路を限定すること
  4. 膨大な量の通信相手を選択すること
  5. 誤りの発生へ対処すること
  6. 通信の秘密をまもること

6. ネットワークの基本概念

6.1 ネットワーク・インターネット・ウェブの違い

  • ネットワーク: 全てがネットワークに繋がっている(と思われている時代)
  • インターネット: 元々は個別のネットワークを相互に繋いだ「インター・ネットワーク」
  • The Internet: 現行のIPv4, IPv6とDNSによる通信網
  • しばしば止まっている、サービスが停止していることもある

6.2 ウェブ・スマホ以外のインターネット応用

6.3 サーバとクライアント

  • クライアント → Request → サーバ
  • サーバ → Service → クライアント

  • 「サーバ」という名称を忌避する傾向(master/slaveの問題、gitでは既に顕著)
  • サーバが常に高機能・高信頼とは限らない
  • 利用者の手元がクライアントとは限らない
X-Windowの例: X-ServerとX-Clientの関係

6.4 プロトコル

  • プロトコル: 外交儀礼
  • RFC(Request For Comment)で規定されたインターネット通信プロトコル
  • 主要プロトコル: HTTP, HTTPS, SMTP, POP, IMAP, NTP, SSH
  • telnet, ftpはよほどのことが無ければ使わない

11. DHCP (Dynamic Host Configuration Protocol)

11.1 基本機能

  • IPアドレスを自動設定する仕組み
  • ルータで動いているため、手動設定不要で通信可能
  • ルータ内で別のDHCPサーバが動作すると混乱の元

11.2 応用

  • Disklessサーバの実現
  • データセンターでの利用:
    • NIM(Network Install Management, IBM)
    • IPMI(Intelligent Platform Management Interface)
    • 電源ON/OFF、UEFI/BIOS、OSインストール、監視が可能
参考事例: 角川グループへのランサムウェア攻撃と関連するインシデント

12. DNS (Domain Name System)

12.1 DNSの仕組み

  • DNS権威サーバ: ドメイン名の管理、ドメイン名とIPの対応を答える
  • DNSキャッシュサーバ: 世界中のDNSルートサーバやDNS権威サーバへ問い合わせ、対応を答える
  • 権威サーバとキャッシュサーバを分離するべき

12.2 DNSキャッシュサーバのIPの取得

  • DHCPから取得
  • DHCPからの応答はブロードキャスト(ローカルエリアネットワーク(L2)全体への通信)

12.3 ドメイン名を使う理由

  1. 記憶しやすさ: 192.168.0.1より www.example.com の方が覚えやすい
    危険: ドメイン名の多言語化による悪用
    例: www.exampⅼe.com と www.example.com (見た目が同じでもUTFのコードが違う)
  2. 柔軟性: IP管理組織を変更してもドメイン名との対応を書き換えれば継続利用可能
    DNSサーバの移転ではしばしば事故が発生
    「浸透言うな!」中京大学鈴木常彦先生の警告

12.4 ルートサーバ

  • a.root-servers.org ~ m.root-servers.org の13種
  • mは日本にある
  • IP anycastにより、実際には複数のサーバが同一IPを使用
  • ネットワーク的に近いサーバが応答して負荷分散
  • 管理の分散も実現

13. NAT (Network Address Translation)

13.1 基本機能

  • プライベートアドレスとグローバルアドレスを変換
  • ルータ上で実現
  • 家庭ではプロバイダから貸与/購入したルータが実現(DHCPも稼働)
  • 変換テーブルで管理

13.2 インターネットの根幹

  • それぞれの機器が対応する変換テーブルを持つ
  • 要求: 早く・大量に・ハードウェアの制約内で

13.3 通信制限

  • プライベートアドレスとグローバルアドレスは直接通信しない
  • ルータが直接通信を遮断
  • プライベートアドレスはインターネットへ流さないことが規定

13.4 (逆)IPマスカレード / Reverse NAT

  • プライベートアドレスでウェブサーバを立てたい場合に使用
  • 用途例: discord, マインクラフトサーバ等
重要な注意: サーバで犯罪(ストーカー等)に関与すると警察が調査・押収に来ます
認証連携していると更に確率UP

14. ネットワークの物理的実体

14.1 Wi-Fi・Ethernetケーブル

  • 無線は盗聴される
    • 対策1: 暗号化
    • 対策2: 接続相手の限定(MACアドレスの利用)
  • Ethernetケーブルの種類:
    • 銅線(より対線)
    • 光ファイバー: 折らない、曲げない、汚さない

14.2 MACアドレス (Media Access Control)

  • 48bit: 上位24bitはベンダーID(製造会社)、下位24bitは識別子
  • スマホにも付いている
  • 匿名性維持のためソフトウェアで変更する例(Apple)
  • 無線LAN基地局のセキュリティと相反する場合がある
なぜMACアドレスで通信しないのか?
  • 機器が変わるとすべての対応表を変更しないとならない
  • 世界中のMACアドレスを管理するメモリ量がハードウェアに対して大きすぎる
  • 注: 千葉工大の2024年8月からのネットワークはこの考え方

14.3 ハブの種類

種類 特徴
リピーターハブ / ダムハブ • 論理的に同軸ケーブルをエミュレート
• 全てのポートに同じ信号が流れる(希少)
• パケットキャプチャできる
• ネットワーク障害の解析に活用
スイッチングハブ • MACアドレスを学習
• 通信しているポート間のみ信号が流れる
• 高速
インテリジェントハブ • モニター機能(SNMP)やVLAN機能を持つ
• 職場では必須
• 勝手にケーブルを付け替えられるとセキュリティホールが発生

14.4 スパニングツリー

  • ブロードキャストストームを防ぐ
  • 全て宛のパケット(Ethernetフレーム)はループになったネットワーク上を回り続ける
  • イーサネットフレームにはTTLが無い
  • 当初は物理的に届くか/届かないかのみを想定
実例: 意図せずケーブルを繋いでしまう人はいる
過去の情報工学科研究室でも発生(無線LAN基地局の勝手な設置等)

14.5 IPパケット

  • Time To Live (TTL)という寿命が設定
  • ルータを通過するたびに1減らされる
  • 0になると捨てられる
  • 理由: パケットが永遠に生き残ると飽和する
途中のルータの数が多いとパケットが届かない

22.2 ロボット三原則

出典: 「われはロボット」2058年の「ロボット工学ハンドブック」第56版より
  1. 第一条: ロボットは人間に危害を加えてはならない。また、その危険を看過することによって、人間に危害を及ぼしてはならない。
  2. 第二条: ロボットは人間にあたえられた命令に服従しなければならない。ただし、あたえられた命令が、第一条に反する場合は、この限りでない。
  3. 第三条: ロボットは、前掲第一条および第二条に反するおそれのないかぎり、自己をまもらなければならない。

22.3 第零法則

  • 第零条: ロボットは人類に危害を加えてはならない。また、その危険を看過することによって、人類に危害を及ぼしてはならない。
  • 第一条の修正: 第零法則に反する場合は第一条の制限が適用されない
